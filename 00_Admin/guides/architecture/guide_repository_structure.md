# Guide: Repository Structure

**Scope:** Repo-agnostic guidance. Replace `<work_repo>` with your target repo root.

## 1. Purpose

This guide defines the **organizational architecture** of a governed target repository. It explains the role and
contents of each top-level folder, the relationships between them, and conventions for file naming and usage. It is
descriptive and instructional -- not a technical specification.

---

## 2. High-Level Layout

```text
00_Admin   -> defines governance, policies, and guides
01_Resources -> shared templates and reference data
02_Modules -> provides reusable capabilities
90_Sandbox -> temporary workbundles and experiments (gitignored)
99_Trash   -> quarantine and purge (gitignored)
```

**Summary:** The ai_ops core distribution ships three tracked directories (`00_Admin`, `01_Resources`,
`02_Modules`) plus two gitignored transient directories (`90_Sandbox`, `99_Trash`) that emerge from
usage. Optional patterns for governed external repos (`01_Agents/`, `Projects/`) are documented in
Section 3.0.1.

  it.

### 3.0.1 Optional Patterns for Governed Repos

The following patterns are **optional** for governed external repos and are not
present in the ai_ops core distribution.

| Pattern | Purpose | Notes |
| --- | --- | --- |
| `01_Agents/` | Orchestration metadata and agent registry | Use only when a repo needs explicit agent registry metadata outside `02_Modules/`. |
| `Projects/` (or `04_Projects/`) | Project-scoped deliverables | Use the suitcase template at `01_Resources/templates/documents/project_directory_template.md`. |

### 3.0.2 Canonical Project Suitcase (Work Repos)

Recommended shape for `<work_repo>/Projects/<project>/`:

```text
admin/
  run_logs/
docs/
inputs/
  data/
  gis_project/
  resources/
    connections/
    layouts/
    styles/
outputs/
  exports/
    pdf/
    shp/
    points/
    reports/
```

Project metadata SHOULD include web-bridge baseline keys:
`project_id`, `storage_prefix`, `crs`, `aoi_source`, `data_contract_version`.

### Database Files (PostgreSQL/PostGIS)

- Never sync the live PostgreSQL `data/` directory or a running PostGIS instance into workspace repositories.
  Keep the service under the default install path (e.g.,
  `C:\Program Files\PostgreSQL\18\data\`) or another local drive.
- Only configuration templates, migration scripts, `.env.example` files, and logical backups created with `pg_dump`
  belong in the repo. Anything that contains real credentials or live database state stays outside version control.
- Document the required environment variables (`POSTGIS_HOST`, `POSTGIS_PORT`, `POSTGIS_DB`, `POSTGIS_USER`) in guides.
  Never commit secrets such as `POSTGIS_PASSWORD`.

### 3.1 00_Admin/db Substructure

The `00_Admin/db/` folder centralizes database-related assets that are safe to keep under version control.

Recommended layout:

```text
00_Admin/
  db/
    migrations/
      0001_init_realestate_gis.sql
      0002_add_parcels_layer.sql
      0003_add_zoning_overlays.sql
    templates/
      database.yml.example
      .env.example
    runbooks/
      rb_postgis_migrations.md
```

Guidelines:

- `migrations/` contains idempotent SQL scripts to create or evolve schemas in databases such as `realestate_gis`.
- `templates/` contains non-secret examples for configuration files that projects or developers can copy and adapt.
- `runbooks/` documents operational procedures (e.g., how to apply migrations, how to restore from backups).
- No file in `00_Admin/db/` should contain real passwords, tokens, or the physical path to a PostgreSQL `data/`
  directory.

### 3.2 GIS layout templates

- GIS layout templates (`layout_*.qpt`) live under `<work_repo>/02_Modules/gis_workflow/templates/layouts/` and are
  treated as import-only templates. Layout behavior and metadata live in Python builders and YAML sidecars; `.qpt`
  files are generated by GIS tools and not edited by hand. Use the `layout_<topic>.qpt` naming pattern from
  `guide_naming_conventions.md`.

### 3.3 Binary reference artifacts

- Binary reference files (e.g., `.docx`, `.xlsx`, vendor PDFs) are allowed when they are source artifacts or vendor
  references.

- Store them under module `docs/` or `examples/` with clear, lowercase filenames.
- Do not use binary formats for new authored guidance or specs when Markdown is feasible.

### 3.4 Core vs Domain Tooling

- Core reusable tooling (scripts, linters) lives in `ai_ops/00_Admin/scripts/`.
- Domain-specific tools (for example GIS checks) belong in
  `<work_repo>/tools/`.
- Tooling docs/manifests should follow the same boundary:
  - core docs in `ai_ops/00_Admin/tooling/`,
  - domain docs in `<work_repo>/00_Admin/tooling/`.

## 7. Adding New Components

### To add a new module

1. Duplicate the folder structure from an existing module.
2. Update the README and `docs/specs/` with a new outline spec.
3. If the repo uses the optional `01_Agents/` pattern, register the module in
   `<work_repo>/01_Agents/metadata/module_graph.yaml` as needed. Otherwise,
   keep module metadata in `<work_repo>/02_Modules/<module>/metadata/module.yaml`.

### To add a new project

1. Start from `01_Resources/templates/documents/project_directory_template.md`.
2. Rename folder to `project_name/`.
3. Ensure canonical project shape (`admin/`, `docs/`, `inputs/`, `outputs/`).
4. Fill `project_metadata.yaml` with baseline contract keys and run-log path.

### To add a new guide

1. Create a file beginning with `guide_` in `/00_Admin/` or an appropriate subfolder.
2. Include YAML front-matter and visible Markdown `# Heading`.
3. Reference it from `00_Admin/guides/architecture/guide_naming_conventions.md` if globally relevant.
