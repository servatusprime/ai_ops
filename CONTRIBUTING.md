# Contributing

## For Humans

See [HUMANS.md](HUMANS.md) for orientation, quick start, and command reference.

### Deployment Model

ai_ops is designed to be installed once and govern multiple project repositories.
ai_ops lives in its own repository. Your project repos are the *work repos*. When
you run `/work` in a project repo, ai_ops governs that session -- work artifacts,
validation, and git operations stay in the project repo. ai_ops stays clean.

This is the primary deployment pattern. Contributions should preserve this
separation: ai_ops does not own project artifacts; project repos do not own
ai_ops governance files.

### How to Contribute

1. **Fork** the repository
2. **Create a branch** (`git checkout -b feature/my-improvement`)
3. **Make your changes** following the authoring rules below
4. **Run validation** (`pre-commit run --all-files`)
5. **Send a pull request** with a clear description of what and why

### Quick Checklist

- [ ] Changes follow `.markdownlint.json` rules
- [ ] New files use `lower_snake_case` naming (plugin surface uses `kebab-case` -- see below)
- [ ] Documentation updated if needed
- [ ] Pre-commit hooks pass
- [ ] If adding or renaming workflow files, regenerate wrappers: `python 00_Admin/scripts/generate_workflow_exports.py`

### Workspace Layout

ai_ops is a standalone repository designed to be installed once and govern
multiple project repositories. In a typical workspace, ai_ops sits alongside
the repos it governs:

```text
workspace/
  ai_ops/        <- this repo
  my_project/    <- a governed work repo
```

All config files (`.markdownlint.json`, `.pre-commit-config.yaml`, etc.) are
self-contained within ai_ops. Contributions should not assume or require
workspace-level config to be present.

## For Agents

For agent runtime behavior, follow `AGENTS.md` Quick Path (`AGENTS.md` first;
load this file on demand for contributor policy and maintainer workflow rules).

Agent runtime governance is consolidated in `AGENTS.md`:

- authority quick reference and path-based guard
- agent decision tree and red flags
- selfcheck loop and completion gate
- approval scope, pre-authorized actions, commit/push gate
- request clarification gate and thriftiness pass

**Primary agent vs subagents:** The agent you interact with directly is the
*primary agent*. In a multi-agent run, the Coordinator (primary agent) spawns
*subagents*, each assigned a specific role (Coordinator, Executor, Builder,
Validator). Subagent definition files live in `plugins/ai-ops-governance/agents/`
and are generated by `/profiles` + `00_Admin/scripts/regenerate_profiles.py`.
Do not hand-edit subagent definition files.

Use this file for contributor policy and maintainer guidance only.

## Architecture Overview

### Organizing Axes

Work flows through five axes: Intent -> Commitment -> Execution -> Verification
(reference flow), with Meta as a cross-cutting governance layer. All artifacts
map to exactly one primary axis. Canonical reference:
`00_Admin/guides/ai_operations/guide_ai_operations_stack.md`

### Plugin Surface

The `plugins/ai-ops-governance/` plugin provides the runtime surface for Claude Code:

| Subdirectory | Contents | Notes |
| --- | --- | --- |
| `commands/` | Command wrappers (`snake_case`) dispatching to `.ai_ops/workflows/` | Thin adapters only |
| `skills/` | Skill wrappers (`kebab-case`) for Claude skills contract compliance | Generated |
| `agents/` | Subagent definition files for multi-agent runs | Generated; do not hand-edit |

Plugin installation:

```sh
claude --plugin-dir ai_ops/plugins/ai-ops-governance
```

Or via marketplace flow: `claude plugin install ai-ops-governance`

Canonical workflow contracts live in `.ai_ops/workflows/`. Wrappers must remain
thin adapters; business logic lives in workflow files, not wrappers.

When adding or renaming workflows, regenerate wrappers:

```sh
python 00_Admin/scripts/generate_workflow_exports.py
```

Validate wrapper sync before committing:

```sh
python 00_Admin/scripts/check_workflow_exports_drift.py --strict --require-manifest
```

### Naming Conventions

| Context | Convention | Reason |
| --- | --- | --- |
| Files and folders | `lower_snake_case` | ai_ops naming policy |
| Plugin `skills/` wrappers | `kebab-case` | Claude skills contract (`[a-z0-9-]+`) |
| Plugin `commands/` wrappers | `snake_case` | Passthrough from workflow source names |
| Profile agent files | `kebab-case` | Generated under skills contract |

Full naming rules: `00_Admin/guides/architecture/guide_naming_conventions.md`

## Design Philosophy

- **Governed separation**: ai_ops governs project repos; it does not own their artifacts.
  One ai_ops install can govern multiple project repos simultaneously.
- **AI-first architecture**: Optimize for agent comprehension and restartability.
- **Context locality**: Place information close to where it is used.
- **Explicit over implicit**: Document rules and placement decisions.
- **Contracts over policies**: Enforce what can be validated.
- **Minimal viable governance**: Add complexity only when evidence demands it.
- **Evolutionary design**: Iterate based on observed patterns.

## For Maintainers

### Environment and Tools

- Clone repo and open in IDE
- Install recommended extensions: `00_Admin/guides/guide_ide_extensions.md`
- Commit/push policy: `00_Admin/policies/policy_git_workflow_conventions.md`

### Authoring Rules

- Markdown: `.markdownlint.json` rules, `.editorconfig` defaults
- Guide: `00_Admin/guides/authoring/guide_markdown_authoring.md`
- Templates: `01_Resources/templates/documents/`

### Naming and Layout

- Use `lower_snake_case` for files/folders (exception: plugin `skills/` -- `kebab-case`)
- Rules: `00_Admin/guides/architecture/guide_naming_conventions.md`
- Structure: `00_Admin/guides/architecture/guide_repository_structure.md`

### Profile and Subagent Management

- Profile source data: `02_Modules/01_agent_profiles/base/default_crew.yaml`
- Per-machine overrides: `.ai_ops/local/profiles/active_crew.yaml` (gitignored)
- Regeneration script: `00_Admin/scripts/regenerate_profiles.py`
- Generated subagent files: `plugins/ai-ops-governance/agents/` (do not hand-edit)
- Primary-agent profile payloads: `02_Modules/01_agent_profiles/generated/lead_agent_profile_context.md`
- Profile management command: `/profiles`

### Workflow Export Drift

After editing any `.ai_ops/workflows/*.md` file:

```sh
python 00_Admin/scripts/generate_workflow_exports.py
```

The pre-commit hook checks for export drift and will block commits if wrappers
are out of sync with workflow sources. Regenerate before committing.

### Pull Requests

- Include or update documentation when needed
- Keep diffs scoped; link related issues
- Tag reviewers per repository reviewer rules (if `CODEOWNERS` is present)
- For Level 4 changes (policies, specs, AGENTS.md, CONTRIBUTING.md, workflows):
  include explicit rationale and gain maintainer approval before merge

### Questions

Open a discussion or issue if anything is unclear.
