# ai-ops-governance Plugin

Governance plugin surface for ai_ops workflows, commands, and subagent profiles.

## Contents

- `.claude-plugin/plugin.json`: Plugin manifest metadata.
- `commands/`: Command wrappers that dispatch to canonical workflow sources.
- `skills/`: Skill wrappers for reusable invocation surfaces.
- `agents/`: Generated functional subagent definitions.

## Installation

This plugin is designed for ai_ops-governed repositories where canonical
workflow contracts live in `.ai_ops/workflows/`.

Recommended Claude surface:

- Claude Code plugin panel (Code tab -> Plugins), not Settings ->
  Capabilities -> Skills zip uploads.

Local development load (from workspace root):

- `claude --plugin-dir ai_ops/plugins/ai-ops-governance`

Project installation (Claude Code plugin marketplace flow):

- `claude plugin marketplace add ./ai_ops/plugins`
- `claude plugin install ai-ops-governance`

Marketplace catalog file:

- `ai_ops/plugins/marketplace.json`

1. Ensure repo setup has been run:
   - `./.ai_ops/setup/setup_claude_skills.sh` (bash)
   - `.ai_ops\setup\setup_claude_skills.bat` (Windows)
2. Regenerate wrappers when workflow contracts change:
   - `python 00_Admin/scripts/generate_workflow_exports.py`
3. Validate wrapper sync and drift:
   - `python 00_Admin/scripts/check_workflow_exports_drift.py --strict --require-manifest`

## Defaults and Customization

- Default behavioral profile: native assistant behavior (no profile override).
- Default crew preset when `/profiles` is used: `balanced`.
- Behavioral tuning path: `/profiles` (rider/crew profile source + regeneration).
- Bootstrap/session preference path: `/customize`.
- Governance-boundary note: policy/spec/workflow architecture changes still
  require upstream approval gates; `/profiles` and `/customize` are not
  governance override lanes.

## Source of Truth

Canonical workflow contracts live in `.ai_ops/workflows/`.
Command and skill wrappers in this plugin should remain thin adapters that
forward execution to those workflow files.

Canonical wrapper generation script:

- `00_Admin/scripts/generate_workflow_exports.py`

Wrapper generation manifest:

- `.ai_ops/exports/manifest.yaml`

## Discovery and Naming Policy

- Canonical workflow files remain `snake_case` in `.ai_ops/workflows/`.
  This preserves ai_ops naming policy and command/workflow identity.
- Plugin `skills/` wrappers are generated as `kebab-case` for Claude
  skills contract compliance (`[a-z0-9-]+`).
- Plugin `commands/` wrappers remain `snake_case` passthrough from the
  source workflow filenames.
- Kebab plugin skill wrappers point back to the `snake_case` workflow
  source files.
- Export naming rationale is recorded in `.ai_ops/exports/manifest.yaml`
  under `naming_policies`.

## Distribution Notes

- Code tab plugin install is the supported distribution path for this repo
  plugin bundle (commands + skills + agents).
- Settings -> Capabilities -> Skills zip uploads are intended for individual
  standalone skills and are not the intended channel for this multi-surface
  plugin bundle.

## Generated Artifacts

Subagent files under `agents/` are generated by:

- `00_Admin/scripts/regenerate_profiles.py`

Lead-agent behavioral profile payloads are generated to:

- `02_Modules/01_agent_profiles/generated/lead_agent_profile_context.md`

This keeps lead profiling adapter-managed and out of root `AGENTS.md`/`GEMINI.md`.

Do not hand-edit generated subagent files. Use `/profiles` or run the
regeneration script after updating profile source data.
